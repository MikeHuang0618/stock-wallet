# 資產歷史 — 核心概念缺口分析與實作規格

> 目的:本工具的定位是「**錢包工具附加看盤功能**」,核心訴求是**紀錄使用者自身資產的歷史**。
> 本文件盤點現況、指出此核心概念仍缺少的能力,並依優先序給出可實作的規格。
> 交給實作者(Opus 4.8)時,請依 Wave 順序執行;每個 Wave 內的項目可獨立交付。

## 0. 現況盤點(2026-07-07,develop @ 9b00df8)

已有的紀錄與計算:

| 能力 | 位置 | 性質 |
|---|---|---|
| 買/賣交易 (side/quantity/price/date) | `wallet.db` `transactions` 表,`api.py:_db` | **落地紀錄** ✅ |
| 出入金 (amount/date/currency) | `wallet.db` `deposits` 表 | **落地紀錄** ✅ |
| 持倉彙總(平均成本法)、已實現損益 | `wallet.py:aggregate_holdings` | 純函式,有 golden test ✅ |
| 錢包價值歷史圖、每日損益 | `wallet.py:build_history` + `api.py:wallet_history` | **每次從 Yahoo 重新回算** ⚠️ |
| USD/TWD 分幣別區塊 + 即時匯率合併 | `wallet.py:summarize_currency` / `combine_currencies` | 合併總計**只有當下值,無歷史** ⚠️ |
| 匯出/匯入 JSON | `api.py:export_data` / `import_data` | 手動觸發 ⚠️ |

**核心矛盾:「資產歷史」目前是衍生檢視 (derived view),不是被記錄的事實 (recorded fact)。**
`wallet_history()` 每次呼叫都重新抓 Yahoo 日線回算。這意味著:

1. 標的下市、改代號、Yahoo 端點失效 → 該標的的歷史**憑空消失**。
2. Yahoo 回補/調整歷史價格(還原權值、拆股) → 使用者「記得的過去」**被改寫**。
3. 股息、手續費、稅金完全沒有紀錄 → 歷史數字**系統性失真**
   (台股高股息 ETF 使用情境下,報酬被顯著低估;已實現損益被高估)。

以下缺口全部圍繞一句話修復:**把「歷史」從回算改為記錄,把「紀錄」補到完整。**

## 1. 缺口清單(依優先序)

優先序 = 價值 × 急迫 ÷ 成本。P0 是核心概念的兌現,不做等於產品承諾不成立。

### P0-1 每日資產快照(歷史從「回算」變「記錄」)

**真問題**:使用者要的是「我的資產在過去每一天值多少」這個**不可變的事實**,
而不是「用今天抓得到的價格重新推算過去」。

**設計**:

- 新表(沿用 `api.py:_db` 的 CREATE-IF-NOT-EXISTS + PRAGMA 遷移模式):

  ```sql
  CREATE TABLE IF NOT EXISTS snapshots(
      date TEXT NOT NULL,            -- YYYY-MM-DD
      currency TEXT NOT NULL,        -- 'USD' | 'TWD'
      market_value REAL NOT NULL,    -- 持倉市值
      cash_balance REAL NOT NULL,    -- 現金餘額(累計出入金 − 累計淨投入)
      portfolio_value REAL NOT NULL, -- 錢包總值
      fx REAL,                       -- 當日 1 USD = ? TWD(可為 NULL)
      detail TEXT,                   -- JSON: [{symbol, qty, close, value}](為除錯與未來配置歷史保留)
      created_at TEXT,
      PRIMARY KEY(date, currency)
  );
  ```

- **寫入時機**:app 啟動與報價背景刷新時,若「今日(或最近收盤日)尚無快照」則寫入;
  同日重複寫入採 UPSERT(收盤前的盤中值會被較新的值覆蓋,收盤後即穩定)。
  不需要常駐排程——這是單機桌面 app,「每次開啟時補寫」即可。
- **讀取策略**:`wallet_history()` 改為「快照優先」:有快照的日期直接用快照;
  快照起點之前的日期沿用現行 `build_history` 回算補齊(向下相容舊用戶)。
  回算段與快照段的銜接日若數值差異 > 1%,在回傳中帶 `reconcile_warning` 供 UI 顯示。
- **不可變紀律**:快照一旦跨日即不再改寫。交易補登(使用者事後補輸入舊交易)時,
  受影響日期範圍的快照標記 `stale`(新增欄位或直接刪除該範圍快照讓回算補齊),
  並在 UI 提示「歷史已依補登交易重算」。**預設絕不靜默改寫。**

**驗收條件**:
- 開啟 app 後,`wallet.db` 中出現當日(每幣別一筆)快照;隔日再開,新增而非覆蓋。
- 把某持倉的 symbol 改成一個 Yahoo 抓不到的假代號後,歷史圖**過去的區段不變**。
- 純計算(快照與回算序列的合併、銜接差異偵測)放 `wallet.py` 純函式,配 golden test。

### P0-2 股息紀錄(現金股利 + 股票股利)

**真問題**:資產歷史的目標使用者(台股 + 美股雙市場)極可能持有配息 ETF。
沒有股息,報酬率永久失真,「紀錄資產歷史」的可信度不成立。

**設計**:

- 沿用單一帳本原則,擴充 `transactions.side` 的合法值:
  - `side='dividend'`(現金股利):`quantity` 存**每股配息金額不拆**——直接 `quantity=1, price=總金額` 太隱晦,
    改為:`quantity` = 股數基準(可為 0 表示直接輸入總額)、`price` = 總金額。
    **實作者請簡化為:`side='dividend'` 時 `price` 即「本次配息總金額」、`quantity` 固定 0**,
    避免兩種輸入模式。UI 表單提供「每股 × 股數」小算盤但落地存總額。
  - `side='stock_dividend'`(股票股利/配股):`quantity` = 配發股數,`price` = 0。
- 數學(改 `wallet.py`,全部純函式 + golden test):
  - `_accumulate_positions`:`dividend` 累加至新欄位 `e["dividends"]`(不動 qty/cost);
    `stock_dividend` 使 `qty += q`、cost 不變(攤低平均成本——這正是配股的正確語義)。
  - `summarize_currency`:`portfolio_value = 存入資金 + 未實現 + 已實現 + 累計股息`;
    報酬率分子同步加入股息。現金股利同時增加 `cash_balance`(概念上等同一筆不減持股的入金)。
  - `build_history` / 快照:配息日起現金餘額 + 股息。
- **選配(可延後)**:抓 Yahoo chart API `events=div` 對持倉標的做「疑似配息提醒」,
  一鍵帶入表單;但**永不自動入帳**——金額以券商實發為準(有稅、有零股差異)。

**驗收條件**:
- 手算 fixture:買 100 股 @50,配息總額 200,股價仍 50 → 未實現 0、累計股息 200、
  報酬率 = 200/投入;配股 10 股 → qty=110、avg_cost=5000/110。golden test 斷言到小數。
- 已實現損益明細與持倉列表各多一欄「累計股息」;錢包總覽顯示「累計股息」totals。
- 舊資料庫(無 dividend 紀錄)行為完全不變(向下相容 test)。

### P1-1 手續費與交易稅

**真問題**:台股賣出有 0.3% 證交稅 + 0.1425% 手續費(常有折扣),不記費用則
已實現損益系統性高估,長期累積後歷史失真。

**設計**:
- `transactions` 加 `fee REAL DEFAULT 0` 欄位(PRAGMA 遷移,同 `side` 前例)。
- 數學:買進 → `cost += q*p + fee`(費用進成本);賣出 → 已實現損益 `-= fee`。
- UI:買賣表單加「費用(含稅)」欄位,預設 0,**不自動套費率**
  (券商折扣人人不同;自動套錯誤費率比留 0 更糟)。可提供台股預估按鈕(0.1425%/賣出加 0.3%)填入,使用者可改。
- 費率常數具名(`TW_BROKER_FEE_RATE = 0.001425` 等),不寫魔法數字。

**驗收條件**:golden test——買 1000 股 @100 費 142,賣 1000 股 @110 費 486
(157 手續費 + 330 稅)→ realized = (110−100)×1000 − 142 − 486(費用進成本法則按實作擇一,
測試與文件說清楚採哪一種,兩處一致)。舊資料 fee 視為 0,數字不變。

### P1-2 股票分割/持股調整 (corporate actions)

**真問題**:美股拆股(如 NVDA 10:1)後,現行帳本的 qty 與均價全錯,且歷史圖在拆股日出現斷崖。

**設計(最小版)**:
- 新 `side='adjust'`:`quantity` = 股數增減(正負皆可)、`price=0`、`note` 說明原因。
  平均成本自動攤薄/濃縮(cost 不變、qty 變)。
- 快照(P0-1)存的是「當日市值」,拆股前的快照本來就正確,**不需回溯調整**——
  這是快照架構的額外紅利,規格書中明說,避免實作者畫蛇添足去改歷史。
- 選配:Yahoo `events=splits` 偵測持倉標的的拆股事件 → 提醒使用者一鍵產生 adjust 交易。

**驗收條件**:100 股 @900 → adjust +900 股 → qty=1000、avg_cost=90、cost_basis 不變、
未實現損益在拆股前後連續(golden test)。

### P1-3 跨幣別合併的「總資產歷史」單一曲線

**真問題**:資產歷史的頭號檢視應該是**一條總淨值曲線**;現在 USD/TWD 歷史分開,
合併總計只有「現在」這一個點。

**設計**:
- 快照表已存每日 `fx`(P0-1)→ 合併歷史 = 逐日 `USD_pv + TWD_pv / fx`。
- 快照之前的回算段:抓 `TWD=X` 日線一次(同 `wallet_history` 的快取機制)做逐日折算;
  某日 fx 缺值沿用前值(forward-fill,與 `build_history` 對股價的處理一致)。
- fx 完全不可得且確有台幣資產 → 該日合併值為 None,前端斷線顯示
  (沿用 `combine_currencies` 的資料新鮮度紀律:**絕不折成 0**)。
- UI:錢包歷史圖新增「總計 (USD)」頁籤或第三條線,預設顯示總計。

**驗收條件**:純函式 `combine_history(usd_hist, twd_hist, fx_series)` 進 `wallet.py`,
golden test 含:fx 缺值 forward-fill、只有單幣別、fx 整段缺失三種情境。

### P2-1 非行情資產(現金以外的手動估值資產)

**真問題**:「錢包」的終點是**淨資產 (net worth)**,不只股票。定存、儲蓄險、
實體黃金、房產、無代號資產目前完全無處記錄,資產歷史只覆蓋了證券部位。

**設計(刻意做小)**:
- 新表 `manual_assets(id, name, currency, created_at)` +
  `manual_valuations(id, asset_id, date, value, created_at)`——
  資產只記「某日估值多少」,不記成本/損益(那是第二期;先解「淨資產有多少」)。
- 快照與歷史曲線把 manual assets 的最新估值(逐日 forward-fill)加進 `portfolio_value`,
  UI 上以獨立區塊列出,配置圓餅圖多一類「其他資產」。
- **明確不做**:手動資產的報酬率、損益、攤銷——估值歷史本身就是價值。

**驗收條件**:新增一筆「定存 100 萬 TWD @2026-01-01」後,總資產曲線自該日起 +100 萬;
更新估值後曲線自更新日反映新值,舊區段不變。

### P2-2 報酬率指標修正:XIRR(年化)取代天真報酬率

**真問題**:現行 `portfolio_return_pct = 總損益 / 總入金` 忽略資金投入的**時間**。
分批入金越久,這個數字越沒有意義;對「紀錄歷史」的工具,時間正是核心維度。

**設計**:
- `wallet.py` 新增純函式 `compute_xirr(cashflows: [(date, amount)], final_value, final_date)`,
  牛頓法 + 二分 fallback;現金流 = 出入金(入金為負、出金為正)+ 期末總值為正。
- UI:錢包總覽的報酬率旁加「年化 (XIRR)」;既有累計報酬率保留但改名「累計損益率(相對入金)」。
- 邊界:持有 < 30 天不顯示年化(數字會荒謬);全負現金流、無解時回 None,UI 顯示「—」。
- **選配(第二期)**:歷史圖疊加 benchmark(0050.TW / SPY)同期定投對照線。

**驗收條件**:golden test 對照 Excel XIRR 至 1e-6:單筆入金一年 +10% → 10%;
兩筆錯期入金案例;無解案例回 None。

### P2-3 自動備份

**真問題**:這個工具的全部價值 = `%APPDATA%\StockWallet\wallet.db` 這一個檔案。
沒有自動備份,一次磁碟事故 = 使用者失去「全部資產歷史」——與核心承諾直接衝突。

**設計**:
- app 啟動時,若當日尚未備份 → `wallet.db` 複製到 `%APPDATA%\StockWallet\backups\wallet-YYYYMMDD.db`,
  保留最近 30 份,超出刪最舊(檔名排序即可)。SQLite 用 `sqlite3` 的 backup API 或
  關閉連線後複製檔案皆可,注意不要在寫入中複製。
- 設定頁加「開啟備份資料夾」與「還原備份…」(選檔 → 關閉現有連線 → 覆蓋前先把現行 db 另存)。

**驗收條件**:啟動兩次(跨日模擬)產生兩份備份;第 31 份出現時最舊一份被刪;
還原後持倉數字等於備份當時。

### P3-1 期間彙總報表(月報/年報)

**真問題**:「歷史」要能回答「我這個月/今年賺賠多少、資產變化多少」;
另外**美股已實現損益按年度彙總(折 TWD)** 是台灣使用者海外所得申報的實際需求。

**設計**:純函式 `summarize_periods(history, transactions, period='monthly'|'yearly')` →
每期:期初值/期末值/淨入金/期間損益/期間報酬率;年度已實現損益表(含股息)按幣別列出,
僅呈現數字,**不做稅務計算**。UI:錢包頁新增「報表」折疊區,表格呈現。

### P3-2 券商對帳單 CSV 匯入

**真問題**:紀錄歷史最大的敵人是**輸入摩擦**——懶得逐筆輸入,歷史就有洞。

**設計**:通用 CSV 匯入(欄位對應 UI:日期/代號/買賣/股數/價格/費用),
預覽 → 確認 → 寫入;以 (date, symbol, side, quantity, price) 全等視為重複並跳過。
不做特定券商格式適配(格式太碎,通用對應 + 預覽已解 80%)。

## 3. 明確不做 (Non-goals)

- 券商 API 即時同步、自動下單——超出「紀錄」定位,且有法遵風險。
- 稅額試算/申報文件——只提供年度彙總數字。
- 雲端同步/多裝置——本工具的隱私承諾是「資料只在本機」(README 明載),不破壞它。
- 選擇權/期貨/融資保證金帳務——平均成本法帳本無法正確表達,勿硬塞。
- 手動資產的損益分析(第一期只記估值)。

## 4. 實作紀律(對實作者的硬性要求)

1. **金錢數學一律進 `wallet.py` 純函式**(不碰 DB/網路),`api.py` 只做 I/O 與委派——
   這是本 repo 的既有架構原則(README「Architecture」節),不得破壞。
2. **每個功能同一個 change 內附 golden test**(手算 fixture 斷言到具體數值,
   不接受只測「不噴例外」);邊界情境(空資料、除以零、缺值 None)必測。
   交付前 `ruff check . && pytest` 全綠,回報通過數。
3. **DB 遷移沿用既有模式**:`PRAGMA table_info` 檢查缺欄 → `ALTER TABLE` 補上,
   舊資料庫開啟後行為不變(每個遷移配向下相容測試)。
4. **缺值紀律**:報價/匯率抓不到一律 None 傳遞到 UI 顯示「—」,絕不用 0 冒充
   (現有 `enrich_holding` / `combine_currencies` 的既定原則)。
5. **命名**:遵循現有慣例——函式動詞開頭(`compute_xirr`)、布林謂語化、
   週期/費率具名參數與常數,不留魔法數字。
6. **匯出/匯入**(`export_data` / `import_data`)需同步涵蓋新表(snapshots、
   manual_assets、fee/dividend 欄位),版本號 +1 並相容舊匯出檔。
